# **1. API**

## **📌 API란?**

🔎 Application Programming Interface

어플리케이션 소프트웨어를 구축하고 통합하기 위한 프로토콜 세트 🎁

간단히 말해서, 정보 제공자와 정보 사용자 간에 정보를 어떻게 주고 받을지에 대한 것을 담은 계약.🔗

사람이 서로 대화하기 위해서는 언어를 사용해야 한다.🗣 또 둘 이상이 모인 사회에서 질서가 유지되기 위해서는 규칙이 필요하다. 👩‍👩‍👧‍👦

🌍컴퓨터들이 모여서 소통하는 네트워크에서도 마찬가지로 데이터를 주고 받기 위해서는 같은 언어와 규칙을 공유해야 한다. 이러한 규칙들을 프로토콜이라 부른다. 

같은 언어를 사용해야 대화가 가능한 것처럼 같은 프로토콜 조합을 사용해야 소통할 수 있다.

컴퓨터 내부에서, 또는 컴퓨터 사이에서 데이터가 교환될 때는 어떤 데이터 형식을 어떤 순서와 절차, 방법을 이용해 교환할 것인지 정해야 한다. 프로토콜은 이에 대해 정의한 규칙들의 집합을 말한다.


날씨 서비스용 API를 예로 들자면, 해당 API는 사용자가 우편 번호를 제공하며 날씨 정보를 요청(requset)📤	하면 생산자는 날씨 정보로 응답(response)📥하도록 지정한다. 

이처럼 API를 사용하면 주고 받는 자원의 검색 방법, 자원의 출처에 대한 지식이 없어도 정보를 주고 받을 수 있다. 이를 통해 제품 또는 서비스가 서로 커뮤니케이션할 수 있으며 애플리케이션 개발을 간소화하여 시간과 비용을 절약할 수 있다. 💸

# **2. RESTful API**

## 📌 **REST란?**

🔎 Representational State Transfer

RESTful API(=REST API)란 REST 아키텍쳐의 제약 조건을 준수하는 웹 API로, 이때 REST(RESTful)은 프로토콜이 아닌, 네트워크 아키텍처 원칙들을 모아놓은 것을 의미한다. 📋 

<br/>

http의 주요 저자 중 한 사람인 Roy fielding이 http 설계의 우수성이 제대로 활용되지 못하는 것이 안타까워 웹의 장점을 최대한 활용할 수 있는 REST를 발표했다. 

REST는 다른 일부 API (ex SOAP, XML_RPC)등이 엄격한 프레임워크를 부과하는 것과 달리 REST API는 거의 모든 언어를 사용하여 개발이 가능하며, 다양한 데이터 포맷을 지원한다. 

예를 들어 데이터 요청이 http를 통해 REST API로 전송되면 REST용으로 설계된 API, 즉 REST API가 HTML, XML, 텍스트, JSON과 같은 다양한 형식으로 메세지를 반환할 수 있다.   

REST는 일종의 가이드 라인📜이지 정해진 명확한 표준이 있는 게 아니기 때문에 개발하는 서비스의 특징, 환경 등을 고려하여 설계되어야 한다.

## 📌 **REST의 구성**
<br/>

REST는 resource(자원), Method(행위), Message(메세지) 세가지로 구성된다. 

"떡볶이에 대한 요리 방법을 책갈피 페이지에 담는다"을 때, POST방식으로 아래와 같은 json을 보내게 된다. 

{

    "recipe":{
        "food": "떡볶이"
    }
} 

이때 요리 방법이 REST의 resource(자원), 책갈피 페이지에 담는 것(POST)가 REST의 verb(행위, method), 떡볶이가 REST의 Message(메세지)에 해당한다.

## 📌 **주요 제약 조건**
<br/>

### 1) **Client-Server 구조**

클라이언트, 서버, 리소스로 구성된 ✌클라이언트-서버 아키텍처✌가 필요하다. 

서버는 API를 제공하고 클라이언트는 사용자 인증, context(세션, 로그인 정보) 등을 관리하는 구조로 역할을 구분하여 각자 개발할 내용을 명확히 하고 서로간의 의존성을 줄인다. 이는 주로 http를 통해 요청을 처리한다
<br/>
<br/>

### **2) Stateless**

세션, 쿠키 등의 상태 정보를 따로 저장하지 않는다.🍪

API 서버는 들어온 request만을 단순하게 처리하며 그 대신 세션 상태에 대한 정보는 클라이언트에 저장한다. 

이를 통해 서비스의 자유도가 높아지고, 서버에서 불필요한 정보를 관리하지 않음으로써 구현이 단순해진다.
<br/>
<br/>

### **3) Cache 가능성**

일부 클라이언트-서버 간 상호작용의 필요성을 제거할 수 있는 캐시 가능 데이터가 필요하다. 

http 웹표준을 그대로 사용하기 때문에 http가 가진 캐싱 기능이 적용된다. 
<br/>
<br/>

### **4) 계층화된 시스템**

![image](https://user-images.githubusercontent.com/96677719/149722859-c6b1d2b7-acb1-413f-b8ad-c25403e13971.png)

추가 계층으로 클라이언트-서버 간의 상호 작용을 조정할 수 있으며 이러한 계층은 로드 밸런싱, 공유 캐시 혹은 보안과 같은 추가 기능을 제공할 수 있다. 🔐
<br/>
<br/>

### **5) self-descriptiveness(자체 표현 구조)** 🛠

REST API 메세지만 보고도 쉽게 이해할 수 있는 자체 표현 구조로 이루어져있다. 

### **6) Uniform Interface**

인터페이스가 균일하다. 

URI로 지정한 리소스에 대한 조작을 통일되고 한정적인 인터페이스로 수행하는 아키텍쳐 스타일을 갖고 있다.

<br/>
<br/>
<br/>

## 📌 **Uniform Interface의 4가지 측면**

REST를 처음 제시한 Roy Fielding은 이를 REST 아키텍쳐 스타일과  다른 네트워크 기반 스타일을 차별화 하는 핵심적 기능이라고 설명했다. 

### **1) 요청에서 리소스 식별**

리소스가 요청에서 식별되며 클라이언트로 반환된 표현으로부터 분리된다.

### **2) 표현을 통한 리소스 조작** 

클라이언트가 리소스를 나타내는 파일을 수신한다. 이 표현에서 조작 또는 삭제를 허용할 수 있도록 충분한 양의 정보가 포함되어야 한다.

### **3) 자기 기술적 (self-descriptive) 메세지** 

클라이언트에 반환되는 각 메세지에 클라이언트가 정보를 어떻게 처리해야 할지 설명하는 정보가 충분히 포함되어야 한다.

### **4) 어플리케이션 상태 엔진으로서의 하이퍼미디어** 

리소스를 할당한 후 REST 클라이언트가 하이퍼 링크를 통해 현재 사용 가능한 기타 모든 작업을 찾을 수 있어야 한다.
