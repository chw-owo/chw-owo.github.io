# 1. API란?

Application Programming Interface

어플리케이션 소프트웨어를 구축하고 통합하기 위한 프로토콜 세트 

간단히 말해서, 정보 제공자와 정보 사용자 간에 정보를 어떻게 주고 받을지에 대한 것을 담은 것. 

날씨 서비스용 API를 예로 들자면, 해당 API는 사용자가 우편 번호를 제공하며 날씨 정보를 요청(requset)하면 셍신자는 날씨 정보로 응답(response)하도록 지정한다. 

이처럼 API를 사용하면 리소스 검색 방법, 리소스의 출처에 대한 지식이 없어도 정보를 주고 받을 수 있다. 이를 통해 제품 또는 서비스가 서로 커뮤니케이션할 수 있으며 애플리케이션 개발을 간소화하여 시간과 비용을 절약할 수 있다.

# 2. RESTful API의 개념

## 1) REST란?

Representational State Transfer

RESTful API란 REST 아키텍처의 제약 조건을 준수하는 웹 API로, 이때 REST(RESTful)은 프로토콜이 아닌, 아키텍처 원칙 세트를 의미한다.  

http의 주요 저자 중 한 사람인 Roy fielding이 http 설계의 우수성이 제대로 활용되지 못하는 것이 안타까워 웹의 장점을 최대한 활용할 수 있는 아키텍처로써 REST를 발표했다. 

웹 서비스, 모바일 어플리케이션 경량화를 위해 만들어진 아키텍처 원칙 세트로, 다른 일부 API (ex SOAP, XML_RPC)등이 엄격한 프레임워크를 부과하는 것과 달리 REST API는 거의 모든 언어를 사용하여 개발이 가능하며, 다양한 데이터 포맷을 지원한다. 

예를 들어 데이터 요청이 http를 통해 REST API로 전송되면 REST용으로 설계된 API, 즉 RESTful API가 HTML, XML, 텍스트, JSON과 같은 다양한 형식으로 메세지를 반환할 수 있다.   

REST는 일종의 가이드 라인이지 정해진 명확한 표준이 있는 게 아니기 때문에 개발하는 서비스의 특징, 환경 등을 고려하여 설계되어야 한다. 


## 2) 주요 제약 조건
- Client-Server 구조: 클라이언트, 서버, 리소스로 구성된 클라이언트-서버 아키텍처가 필요하다. 서버는 API 제공, 클라이언트는 사용자 인증, context(세션, 로그인 정보) 등을 관리하는 구조로 역할을 구분하여 각자 개발할 내용을 명확히 하고 서로간의 의존성을 줄인다. 이는 주로 http를 통해 요청을 처리한다

- Stateless: 세션, 쿠키 등의 상태 정보를 따로 저장하지 않는다. API 서버는 들어온 request만을 단순하게 처리하며 그 대신 세션 상태에 대한 정보는 클라이언트에 저장한다. 이를 통해 서비스의 자유도가 높아지고, 서버에서 불필요한 정보를 관리하지 않음으로써 구현이 단순해진다.

- Cache 가능성: 일부 클라이언트-서버 간 상호작용의 필요성을 제거할 수 있는 캐시 가능 데이터가 필요하다. http 웹표준을 그대로 사용하기 때문에 http가 가진 캐싱 기능이 적용된다. 

- 계층화된 시스템: 추가 계층으로 클라이언트-서버 간의 상호 작용을 조정할 수 있으며 이러한 계층은 로드 밸런싱, 공유 개시 혹은 보안과 같은 추가 기능을 제공할 수 있다. 

- self-descriptiveness(자체 표현 구조): REST API 메세지만 보고도 쉽게 이해할 수 있는 자체 표현 구조로 이루어져있다. 

- Uniform Interface: 인터페이스가 균일하다. RESTful API의 핵심 URI로 지정한 리소스에 대한 조작을 통일되고 한정적인 인터페이스로 수행하는 아키텍쳐 스타일

## 3) Uniform Interface의 4가지 측면

REST를 처음 제시한 Roy Fielding은 이를 REST 아키텍쳐 스타일과  다른 네트워크 기반 스타일을 차별화 하는 핵심적 기능이라고 설명했다. 

- 요청에서 리소스 식별: 리소스가 요청에서 식별되며 클라이언트로 반환된 표현으로부터 분리된다.

- 표현을 통한 리소스 조작: 클라이언트가 리소스를 나타내는 파일을 수신한다. 이 표현에서 조작 또는 삭제를 허용할 수 있도록 충분한 양의 정보가 포함되어야 한다.

- 자기 기술적 (self-descriptive) 메세지: 클라이언트에 반환되는 각 메세지에 클라이언트가 정보를 어떻게 처리해야 할지 설명하는 정보가 충분히 포함되어야 한다.

- 어플리케이션 상태 엔진으로서의 하이퍼미디어: 리소스를 할당한 후 REST 클라이언트가 하이퍼 링크를 통해 현재 사용 가능한 기타 모든 작업을 찾을 수 있어야 한다. 


## 4) 장점

경량화되어 있어서 IoT, 모바일 어플리케이션 개발, Serverless 컴퓨팅처럼 새로운 컨텍스트에 이상적
